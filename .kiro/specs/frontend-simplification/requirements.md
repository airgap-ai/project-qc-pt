# 前端设计稿简化需求文档

## 简介

根据初始需求文档，当前前端设计稿包含了许多超出核心需求的功能。本文档定义了简化方案，确保在有限的开发周期内聚焦核心功能，降低开发风险。

## 术语表

- **项目质控系统**：基于 AI 的数字化项目辅助工具
- **内容查重**：检测文档与历史项目库的重复内容
- **内容审核**：检查文档的规范性和质量
- **项目质控**：监控项目生命周期关键节点和风险预警
- **AI 校对工作台**：人工审核 AI 提取结果的界面
- **简化版**：仅保留核心功能的最小可行产品（MVP）

## 需求

### 需求 1：保留核心页面结构

**用户故事**：作为项目管理人员，我希望能够快速访问核心功能，以便高效完成日常工作。

#### 验收标准

1. 当用户访问系统时，系统应当展示简洁的导航结构，包含项目列表、任务中心和基础配置
2. 当用户点击导航菜单时，系统应当快速切换到对应页面
3. 当页面加载时，系统应当在 2 秒内完成渲染
4. 系统应当保留统一的设计风格和交互模式

### 需求 2：优化项目详情页面的阶段展示

**用户故事**：作为项目管理人员，我希望在查看当前阶段详情时，也能够查看之前已完成阶段的状态和文档，以便全面了解项目进展。

#### 验收标准

1. 当用户点击项目卡片时，系统应当根据项目当前阶段打开对应的详情页
2. 当用户在阶段详情页时，系统应当在阶段进度条上显示所有阶段的完成状态
3. 当用户点击阶段进度条上的已完成阶段时，系统应当展开该阶段的文档清单和校验结果（折叠面板或侧边栏）
4. 当用户查看历史阶段时，系统应当以只读模式展示文档和校验结果
5. 系统应当保留四个独立的阶段详情页（立项、招标、实施、验收），但增强历史阶段查看功能

### 需求 3：保留文档上传和任务管理

**用户故事**：作为项目管理人员，我希望能够上传文档并查看 AI 处理任务的状态，以便跟踪工作进度。

#### 验收标准

1. 当用户访问任务中心时，系统应当展示所有待处理和已完成的任务
2. 当用户上传文档时，系统应当显示上传进度和任务创建确认
3. 当 AI 处理完成时，系统应当在任务列表中更新状态
4. 当任务需要人工处理时，系统应当提供"去校对"按钮跳转到校对工作台

### 需求 4：保留 AI 校对工作台核心功能

**用户故事**：作为审核人员，我希望能够查看 AI 提取的结果并进行人工确认或修改，以确保数据准确性。

#### 验收标准

1. 当用户打开校对工作台时，系统应当展示文档预览和提取字段列表
2. 当用户点击字段时，系统应当在文档中高亮显示对应位置
3. 当用户修改字段值时，系统应当实时更新并移除错误标记
4. 当用户保存或驳回时，系统应当提交数据并返回任务中心

### 需求 5：大幅简化规则配置功能

**用户故事**：作为系统管理员，我希望能够配置基础的校验规则，但不需要复杂的模板编辑功能。

#### 验收标准

1. 系统应当删除独立的模板编辑页面（template-edit.html）
2. 当用户访问规则配置时，系统应当仅展示简化的规则列表
3. 当用户添加规则时，系统应当提供简单的表单，包含规则名称、类型和基本参数
4. 系统应当删除复杂的阶段配置、动态周期、跨文档比对等高级功能
5. 系统应当保留基础的关键词检测、签字检测规则配置

### 需求 6：删除新建项目功能

**用户故事**：作为开发团队，我们希望在初期版本中专注于核心的查重、审核和质控功能，而不是项目创建。

#### 验收标准

1. 系统应当删除仪表板中的"新建项目"按钮和模态框
2. 系统应当假设项目数据由后端系统或其他模块创建
3. 系统应当仅展示已存在项目的列表和详情

### 需求 7：简化查重报告展示

**用户故事**：作为审核人员，我希望能够查看文档的查重结果，包括相似度和重复片段。

#### 验收标准

1. 当文档完成查重分析时，系统应当在项目详情页展示查重报告入口
2. 当用户点击查重报告时，系统应当展示相似度百分比和重复内容列表
3. 当用户查看重复片段时，系统应当高亮显示原文档和参考文档的对应位置
4. 系统应当支持同步滑动浏览两个文档

### 需求 8：保留质量评分和修改建议

**用户故事**：作为审核人员，我希望看到文档的质量评分和具体的修改建议，以便快速定位问题。

#### 验收标准

1. 当文档完成审核时，系统应当在校对工作台展示质量评分
2. 当存在规范性问题时，系统应当列出具体的错误位置和修改建议
3. 当用户点击建议时，系统应当在文档中高亮显示对应位置
4. 当用户采纳建议时，系统应当自动更新字段值

### 需求 9：保留风险预警功能

**用户故事**：作为项目管理人员，我希望能够及时收到项目延期或合规风险的预警通知。

#### 验收标准

1. 当项目节点临近或逾期时，系统应当在项目详情页显示预警标识
2. 当存在关键文档缺失时，系统应当在文档清单中标注
3. 当用户查看预警详情时，系统应当展示具体的风险描述和建议措施
4. 系统应当在仪表板的项目卡片上显示预警状态（红/黄/绿灯）

### 需求 10：删除不必要的文档和演示功能

**用户故事**：作为开发团队，我们希望清理不必要的演示文档，保持代码库整洁。

#### 验收标准

1. 系统应当删除 DEMO-GUIDE.md、MODAL-INTERACTION-GUIDE.md 等演示文档
2. 系统应当删除 IMPLEMENTATION-SUMMARY.md、INTERACTION-ENHANCEMENTS.md 等开发日志
3. 系统应当保留 README.md 和快速开始.md 作为核心文档
4. 系统应当更新 README.md，反映简化后的功能范围

## 需求边界与约束

- 简化工作应当在不破坏现有设计系统的前提下进行
- 所有保留的功能应当保持原有的交互体验和视觉风格
- 删除的功能应当确保不影响核心业务流程
- 简化后的系统应当能够在 2-3 周内完成后端对接和测试

## 开发规划

本次简化为前端设计稿优化任务，重点是：
1. 删除超出核心需求的页面和功能
2. 合并重复的页面结构
3. 简化复杂的配置界面
4. 清理不必要的文档

后续开发将基于简化后的设计稿进行后端对接和功能实现。
